# ctx 1.2.0

ctx 1.2.0 is an upgrade from ctx 1.1.0. It improves configuration and project management, changes how workspace identifiers are stored, and adds a command-log API for add-ons.

## What's New

- Added `ctx config` and the `xconfig` shell shortcut for managing ctx configuration.
- Added `ctx config get`, `ctx config set`, and `ctx config ls` for the `project.root` setting.
- `ctx project ls` now displays project IDs and names.
- `ctx project rm` accepts either a project ID or name.
- `ctx workspace ls` now displays workspace IDs, names, and paths with a header.
- Added `ctx log start` and `ctx log finish` JSON endpoints for add-ons to create and complete command logs without accessing the ctx database directly.

## Workspace and Database Changes

- Workspace records now use internal integer IDs in the database.
- Workspace names and root paths are stored separately, with the root path remaining unique.
- Existing workspace data directories continue to use UUID-based directory names.
- Related records, including targets, credentials, services, notes, and command logs, are migrated to the new workspace IDs.

## Upgrade Notes

- Existing ctx 1.1.0 databases are migrated automatically when ctx opens them.
- No manual migration command is required.
- Existing workspace data directories are preserved during the migration.
- Existing daily workflows remain available, including `pj root`, `pj <name>`, and `pj rm <name>`.

## For Users

- Use `ctx config get project.root` to inspect the configured project root.
- Use `ctx config set project.root <path>` to change it.
- Use `ctx project ls` to find a project ID before removing a project.
- Use `ctx project rm <id>` when a name may not uniquely identify the intended project.

## For Add-ons

Add-ons can use the versioned JSON log API instead of accessing the ctx database directly:

```sh
ctx log start --format json --format-version 1
ctx log finish <id> --format json --format-version 1
```

The start and finish request bodies are read from standard input, and the response is written as JSON to standard output.
