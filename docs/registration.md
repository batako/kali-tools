# ctx Registration Commands

This document defines the existing human-facing commands that custom and official tools may use to register shared ctx data. These are CLI integration contracts, not JSON APIs.

Use the [ctx Integration Guide](integration.md) to choose an integration method. Use [ctx JSON API](api.md) when structured output is required.

## Contract Scope

For the command forms documented here, integrations may depend on:

- command and option names;
- argument order and meaning;
- the documented duplicate behavior;
- exit code `0` for success; and
- a nonzero exit code for invalid input, missing context, conflicts, or persistence failures.

The current normal-command failure code is `1`. Unlike JSON endpoints, these commands do not distinguish invalid arguments with exit code `2` and do not return machine-readable error codes.

Integrations must not depend on:

- exact stdout or stderr text;
- whitespace, punctuation, colors, or line ordering;
- an ID embedded in a human-readable success message; or
- internal database IDs remaining stable after deletion and recreation.

Successful commands write human-readable confirmation to stdout. Failures write a human-readable diagnostic to stderr. Treat both as messages for the user, not data to parse.

## Prerequisites

All registration commands require an active workspace.

```sh
ctx workspace init
```

Target, host, and credential registration also require target context as described below. Notes require only the active workspace.

## Target Context

Targets establish the identity to which hosts, services, credentials, and target-scoped findings belong.

### Set or update the primary target

```sh
ctx target set <ip>
ctx target update <ip>
ctx target <ip>
```

`set`, `update`, and the shorthand currently perform the same operation:

- the IP must be a valid IPv4 or IPv6 address;
- if no target exists, a primary target named `default` is created;
- if a primary target exists, its IP is updated without replacing its target identity; and
- records associated with that target remain associated after the IP changes.

Running the command again with the same IP succeeds.

### Add another target

```sh
ctx target add <ip> [--name <name>]
```

- the IP must be valid;
- an omitted name is generated by ctx;
- the first target in a workspace becomes primary; and
- a conflicting target name fails instead of silently selecting or replacing another target.

Select an existing target as primary with:

```sh
ctx target use <name>
```

Target removal is destructive and is not a registration operation.

## Hosts

Register a hostname against the primary target:

```sh
ctx host add <hostname>
ctx host <hostname>
```

Register it against a named target:

```sh
ctx host add <hostname> --target <target-name>
```

Behavior:

- an active workspace is required;
- the primary target is required when `--target` is omitted;
- the named target must exist when `--target` is used;
- surrounding whitespace and one trailing dot are removed;
- the hostname is stored in lowercase;
- labels must be nonempty and at most 63 characters;
- the total hostname must be at most 253 characters;
- letters, digits, hyphens, and underscores are accepted;
- labels may not begin or end with a hyphen; and
- whitespace, `/`, `\`, and `:` are rejected.

The key is target plus normalized hostname. Registering the same hostname for the same target again is idempotent: it succeeds and refreshes the record as a manual registration. The same hostname may be associated with another target.

## Credentials

Credential identity is the current primary target plus `scope` plus `username`.

### Create or replace

```sh
ctx credential set <scope> <username> [password]
ctx credential <scope> <username> [password]
```

`set` and the shorthand create the credential when absent and replace its password when present.

### Create only

```sh
ctx credential add <scope> <username> [password]
```

`add` fails when the same target, scope, and username already exist. Use `set` when idempotent create-or-replace behavior is required.

### Update only

```sh
ctx credential update <scope> <username> [password]
```

`update` fails when the credential does not exist.

For all three forms:

- an active workspace and primary target are required;
- scope and username must contain at least one non-whitespace character;
- scope and username are stored as provided and are not case-normalized; and
- omitting the password stores a missing password, and therefore clears an existing password when used with `set` or `update`.

### Secret handling

The current command accepts a password only as an argument. This may expose it through shell history, the process list, terminal recording, or the human-readable success output.

Do not invoke this form with a secret from an unattended integration unless that exposure is acceptable in the environment. Do not parse or persist the success output. A safe JSON-stdin registration form is not currently implemented and must be designed only when a concrete integration requires it.

## Notes

Append a note to the active workspace timeline:

```sh
ctx note <text...>
```

Behavior:

- all remaining arguments are joined with single spaces;
- leading and trailing whitespace is removed;
- empty notes fail;
- notes are workspace-scoped rather than target-scoped; and
- duplicate note bodies are allowed and create separate timeline entries.

If preserving exact whitespace, newlines, or arbitrary binary content is required, `ctx note` is not a suitable machine interface.

## Unsupported Registration Types

There is currently no generic public registration command for:

- services and scan records;
- web discoveries;
- web wordlist run history;
- arbitrary database records; or
- tool-private progress caches.

Their internal Go functions and database tables are not public registration contracts. Keep unsupported results in the producing tool until a concrete use case justifies a validated ctx command or machine-readable input.

## Caller Rules

A maintained integration should:

1. invoke the installed ctx executable without a shell;
2. pass each argument separately;
3. treat exit code `0` as success and any other code as failure;
4. show stderr to the user without matching its text;
5. avoid reading IDs or other values from human-readable stdout; and
6. avoid passing secrets in argv when the exposure is unacceptable.
