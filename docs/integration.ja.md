# ctx連携ガイド

`ctx`は、THMなどのセキュリティラボで必要になるすべてのツールを実装することを目指しません。代わりに、独立したコマンドから現在の調査コンテキストと保存済みの調査結果を再利用するための連携手段を提供します。

この文書では、必要な連携レベルの選び方を説明します。特定のプログラミング言語、SDK、アドオンフレームワークは要求しません。

## 連携レベルの選択

| 必要なこと | 連携方法 | 適した用途 |
|---|---|---|
| ctxのデータを使わない | 独立したコマンド | 現在のターゲットや保存済み結果が不要 |
| 既存コマンドを実行して記録する | `ctx x` | プロセス全体のログとターゲットIP展開だけで十分 |
| コンテキストや調査結果を参照する | 通常のctxコマンドとJSON出力 | ターゲット、サービス、credentialなどが必要 |
| 調査結果を保存する | 既存のctx登録コマンド | `ctx host add`など対応するコマンドが存在する |
| コマンドログを制御する | `ctx log start/finish` | 記録するコマンド、出力、状態、時刻を独自に制御したい |
| 未対応の集計や試作を行う | SQLite直接参照 | スキーマ互換性が不要 |

必要な能力を満たす中で、ctxへの依存が最も少ない方法から選びます。簡単な方法で足りない場合だけ、JSONや構造化ログへ進みます。

## 独立したコマンド

ctxのデータが不要なコマンドは、完全に独立したまま実装できます。ctxのパッケージ、データベース、命名規則、インストール方法に合わせる必要はありません。

ctxへの依存がないため、最も壊れにくい方法です。

## `ctx x`で既存コマンドを実行する

既存コマンドを現在のターゲットへ実行し、workspaceのログへ記録するだけなら`ctx x`を使用します。

```sh
ctx x nmap -sV '$IP'
```

現在の`ctx x`は次を行います。

- 各引数内の`$IP`と`${IP}`をプライマリターゲットIPへ展開する
- stdinを子プロセスへ渡す
- stdoutとstderrを端末へ表示する
- コマンド、展開後コマンド、出力、状態、終了コードをctxログへ保存する
- 子プロセスの終了コードを返す

呼び出し元のシェルが先に展開しないよう、`$IP`をクォートしてください。

### 引数とTarget展開

`ctx x`はshellを起動せず、指定された実行ファイルを直接起動します。`ctx x`より後の各引数は、分離した子プロセス引数として渡します。pipe、redirect、変数展開などのshell演算子は、`ctx x sh -c '...'`のように呼び出し側が明示的にshellを実行しない限り解釈しません。

引数内の`$IP`と`${IP}`は、出現箇所ごとにPrimary TargetのIPへ置換します。ctxが展開するのはこの2つのplaceholderだけです。いずれかを含む場合はPrimary Targetが必要ですが、placeholderを含まないコマンドはTargetなしでも実行できます。

### 入出力とログ

子プロセスへ呼び出し元のstdinを渡します。stdoutとstderrはctxの対応するstreamへリアルタイムに出力しながら、command logにも保存します。ctxは子プロセス起動前に`running`ログを作成し、子プロセス終了後に完了状態へ更新します。

ログには元のコマンド、展開後コマンド、stdout、stderr、開始・終了時刻、状態、終了コードを保存します。引数や出力に秘密情報が含まれる場合、その内容も記録されるため、記録してはいけない値を扱うコマンドへ`ctx x`を使用しないでください。

ログを開始できなければ子プロセスは実行しません。子プロセス完了後のログ保存に失敗した場合は、子プロセスが完了済みでもctxを失敗として扱います。

### 終了と中断

通常終了では子プロセスの終了コードをそのまま返します。終了コード`0`は`success`、それ以外の通常終了は`failed`として記録します。

実行ファイルを起動できない場合は`127`を返し、stderrへエラーを表示して、同じ診断を含む`failed`ログを保存します。子プロセスがsignalで終了した場合は`128 + signal`（SIGTERMなら`143`）を返し、終了コードなしの`interrupted`ログとして保存します。

選択したサービスやcredentialが必要な場合、または解析した結果を保存する場合は別の連携方法を使います。

## 調査情報を参照する

値が1つだけ必要な場合は、単一フィールドを出力するコマンドを使います。

```sh
ctx prompt --field target-ip
ctx prompt --field workspace-path
```

複数の値、一覧、nullを取り得る値、人間向けテーブルが変更される可能性のあるデータはJSONで取得します。

```sh
ctx prompt --format json --format-version 1.0
ctx service ls --format json --format-version 1.0
ctx credential ls ssh --format json --format-version 1.0
```

利用可能なJSON出力とバージョンを事前に確認できます。

```sh
ctx formats --format json --format-version 1.0
```

`data.formats`の各キーを、対応している連携機能として扱います。現在定義されている機能キーは次のとおりです。

| 機能キー | 操作 |
|---|---|
| `formats` | 対応する機能キーとフォーマットバージョンを確認する |
| `prompt` | 現在のWorkspaceとTargetのコンテキストを取得する |
| `credential` | 保存済みcredentialを一覧取得する |
| `service` | 保存済みserviceを一覧取得する |
| `log` | 構造化コマンドログを開始・完了する |

必要な機能を使う前に、そのキーが存在し、バージョン配列に実際に要求する完全なフォーマットバージョンが含まれることを確認します。キーまたはバージョンがない場合、終了コードが0以外の場合、`success: false`の場合は、その機能を利用できません。ctxパッケージのバージョンから機能対応を推測しないでください。

既存の`formats`出力を機能確認インターフェースとし、別のfeature endpointは追加しません。

人間向けの表を`grep`、`awk`、列位置、正規表現などで解析しないでください。文言、空白、列、並び順、装飾は外部連携仕様の対象ではありません。

credentialのJSONには平文パスワードが含まれます。ログ、コマンド引数、診断出力、プロセス一覧、永続的な一時ファイルへ露出させないでください。

レスポンスの外枠、フィールド、エラー、終了コード、フォーマットバージョンは[ctx JSON API](api.ja.md)を参照してください。

## 言語非依存の連携手順

外部連携で利用するのは、特定のライブラリやプログラミング言語ではなく、ctxのコマンドラインインターフェースです。プロセスを起動し、引数を分離して渡し、標準入力へ書き込み、JSONを解析できる任意の言語で次の手順を実装できます。

1. 信頼できるctx実行ファイルの絶対パスを選びます。現在のAPT版は`/usr/local/bin/ctx`です。shell command文字列を組み立てず、実行ファイルと引数配列を分離して直接起動します。

2. カスタムコマンドが必要とするAPIの完全なフォーマットバージョンを決めます。ctxパッケージのバージョンから推測してはいけません。

3. `formats`を呼び出し、必要なすべての機能について`data.formats`に完全一致するversionが含まれることを確認します。機能またはversionがなければ非対応として扱います。

4. 必要なctx操作を`--format json --format-version <version>`付きで呼び出します。標準出力、標準エラー出力、終了コードを個別に取得します。JSONリクエストが必要な操作では、引数へ埋め込まず、1つのJSON値へ変換して標準入力から渡します。

5. 標準出力を1つの共通JSONレスポンスとして解析します。標準エラー出力や人間向けの表をデータとして解析してはいけません。標準エラー出力は診断用であり、空の場合もあります。

6. `format_version`が要求した完全なバージョンと一致することを確認します。`success: true`では、利用前に文書化された`data`の構造を検証します。null、空配列、フィールド欠落を区別します。

7. `success: false`では`error.code`で分岐し、未知の下位コードの場合だけ上位コードを使用します。`error.message`と`error.details`は診断用であり、分岐条件には使いません。不正なリクエストは通常`2`、対象なしと実行失敗は通常`1`で終了します。

8. 標準出力が不正なJSON、成功レスポンスなのに終了コードが0以外、レスポンスのバージョンが不一致の場合は、連携形式のエラーとして扱います。0以外で終了しても有効な失敗レスポンスを取得できた場合、機械判定にはその`error.code`を使用します。

9. 調査結果に対応する文書化済み登録コマンドがあれば、それを使って保存します。人間向けメッセージではなく、構文と終了コードを判定に使います。未対応の結果をSQLiteへ直接書き込まず、カスタムコマンド側で管理します。

10. カスタムコマンドが構造化ログを管理する場合は、外部処理前にログを開始し、返されたログIDを保持し、同じIDを最終状態で完了します。ログ開始に失敗した場合は外部処理を開始しません。

JSON参照の言語非依存擬似コード:

```text
ctx_path = trusted_absolute_path()
required_version = "1.0"

capabilities = run_process(
  executable = ctx_path,
  arguments = ["formats", "--format", "json", "--format-version", required_version],
  stdin = empty
)
formats_envelope = parse_and_validate_envelope(capabilities)
require formats_envelope.data.formats["prompt"] contains required_version

response = run_process(
  executable = ctx_path,
  arguments = ["prompt", "--format", "json", "--format-version", required_version],
  stdin = empty
)
prompt_envelope = parse_and_validate_envelope(response)
require prompt_envelope.success
use prompt_envelope.data
```

JSONの標準入力が必要な場合も順序は同じで、`stdin`だけを空からJSONへ変換済みのバイト列へ変更します。credentialのJSON、パスワード、トークン、Cookieを診断ログへ含めてはいけません。

## 調査結果を保存する

保存する調査結果を既存のctxコマンドで表現できる場合は、そのコマンドを使用します。

```sh
ctx host add admin.example.thm
ctx credential add ssh root '<password>'
ctx note 'admin.example.thm redirects to the management portal'
```

機械連携では、文書化されたコマンド構文と終了コードだけを判定に使います。人間向けの成功・エラーメッセージは解析しません。

調査結果保存専用の汎用APIはありません。また、すべての内部レコードを書き込む汎用公開コマンドもありません。具体的な調査結果をctxで表現できない場合は、実際の連携事例から新しいctxコマンドまたは機械入力が必要だと確認できるまで、カスタムツール側で管理します。

パスワードを引数へ渡すと、シェル履歴やプロセス一覧に露出する可能性があります。この例は既存コマンドの動作を示すものであり、無人処理で安全に秘密情報を渡せることを保証しません。具体的な連携で必要になった時点で、対象コマンドへ安全な機械入力を追加します。

ctxのコマンド形式と現在の動作は[ctxコマンドドキュメント](commands/ctx.ja.md)に記載します。

## 構造化ログを制御する

`ctx x`では制御が足りない場合に`ctx log start/finish`を使用します。例えば、対話型アドオンが展開後コマンドを無害化したり、出力を自分で収集したりする場合です。

JSONを標準入力から渡してログを開始します。

```sh
printf '%s\n' '{"command":"custom-scan","expanded_command":"scanner 10.10.10.10","started_at":"2026-07-19T00:00:00Z"}' | \
  ctx log start --format json --format-version 1.0
```

返されたログIDを取得し、処理結果を渡して終了します。

```sh
printf '%s\n' '{"status":"success","exit_code":0,"stdout":"","stderr":"","ended_at":"2026-07-19T00:01:00Z"}' | \
  ctx log finish 1 --format json --format-version 1.0
```

パスワード、認証用コマンド引数、トークン、Cookieをログのフィールドへ含めないでください。ライフサイクルの形式は[ctx JSON API](api.ja.md#log)を参照してください。

## SQLiteを直接参照する

SQLite直接参照は、試作、未対応の集計、高度なローカル解析に利用できます。継続的に保守するカスタムコマンドの互換境界としては推奨しません。

DBスキーマはctxの内部実装であり、ctxの更新で変更される可能性があります。直接書き込みは入力検証、関連データの更新、トランザクションを迂回し、workspaceを破損させる可能性があります。実験前にバックアップし、読み取り専用での利用を優先してください。

現在のスキーマとマイグレーション方針は[データベース設計](database.ja.md)を参照してください。

## データ所有範囲

ここでいう所有とは、データ形式、入力検証、マイグレーション、削除までのライフサイクルに責任を持つコンポーネントを示します。保存場所だけでは決まりません。ctxのworkspace配下に保存されたファイルでも、特定ツール専用のキャッシュなら、そのツールが所有します。

### ctxが所有するデータ

ctxは、複数のコマンドから共有する永続的な調査コンテキストと調査結果を所有します。

| データ | ctxの責任 | 生成・利用するものの例 |
|---|---|---|
| Workspaceの識別情報とメタデータ | 現在のworkspaceの解決、同一性の維持、DBマイグレーション | すべての連携コマンド |
| Project rootとproject・workspaceの関係 | 名前付きrootの保存、active rootの選択、project名の検証、projectディレクトリとworkspaceの関連付け | `ctx project`、シェル連携 |
| Target | targetの同一性、プライマリ選択、IP変更、関連レコードの維持 | `ctx target`、スキャナ、接続ツール |
| Host | ホスト名の検証とtargetへの関連付け | 手動登録、DNS・vhost探索ツール |
| Serviceとスキャン履歴 | port、protocol、product、version、スキャン根拠の正規化と保存 | `ctx scan`、serviceを使うアドオン |
| Credential | scope付きusername・passwordのtargetへの関連付けと検証根拠の保持 | 手動登録、認証ツール、接続ツール |
| Note | workspace単位の人間向け調査結果をタイムラインへ保存 | 利用者、noteを生成するツール |
| Command log | コマンドのライフサイクル、無害化したコマンド、出力、状態、終了コードの保存 | `ctx x`、`ctx log`を使うアドオン |
| 共有Web探索結果 | path、レスポンス情報、source tool、wordlist、logとの関係をtarget単位で保存 | Web探索、`xweb`での結果閲覧 |
| 共有Webワードリスト実行履歴 | 対応する探索コマンド間で完了・中断した検索を共有 | `xgobuster`、`xffuf` |
| ワードリストカタログと推薦順 | 利用可能性、用途、適合度、優先度、既知ファイルの検証を一元管理 | `ctx wordlist`、`xffuf`、`xgobuster`、`xhydra` |
| 設定の保存 | 設定値の保存、入力検証、デフォルト値の提供 | ctx、文書化されたキーを使うツール |

共有データを生成したツールが、そのデータの所有者になるわけではありません。例えばvhost探索ツールがhostを登録しても、ホスト名の検証、targetへの関連付け、永続化、マイグレーションはctxが担当します。

ctxが所有するのは名前付きproject root設定、active rootの選択、workspace marker、projectとworkspaceの関係です。利用者のprojectファイルやprojectディレクトリ内に作成された任意のファイルまでは所有しません。

ctx所有データのすべてに公開連携コマンドやJSON endpointが存在するわけではありません。特にWeb探索結果やワードリスト実行履歴の内部機能は、それだけで公開APIになるものではありません。カスタムコマンドは、公開Go関数やDBテーブルから安定した連携仕様を推測してはいけません。具体的な連携で必要になった時点で公開操作を追加します。

同梱ツールの自動ワードリスト選択はctxの共通推薦処理を直接利用します。外部コマンドから候補を取得する場合は、独自に`/usr/share/wordlists`を走査せず、`ctx wordlist --kind <kind> --format json`を使用します。

### 各ツールが所有するデータ

外部コマンドやカスタムコマンドは、そのツールだけに意味がある実装データを所有します。

| データ | ツールの責任 | 例 |
|---|---|---|
| 外部コマンドの組み立て | 実行ファイル、フラグ、引数、環境変数の選択 | Gobuster、Hydra、ffuf、SSHの実行 |
| 出力の解析と分類 | ツール固有出力の解釈と調査結果の判定 | HTTPレスポンスの除外、Hydraの成功判定 |
| 探索戦略 | ctxが推薦したwordlistの消費、リクエスト上限、escalation、retryの選択 | directory、vhost、password、username探索 |
| 再生成可能な進捗キャッシュ | 専用キャッシュ形式の定義、更新、削除、移行または破棄 | 検索済みwordファイル、ツール別完了状態 |
| 一時ファイル | 安全な作成と利用後の削除 | 間引いたwordlist、機械可読な結果ファイル |
| 対話動作 | prompt、選択順、default、端末制御、補完 | service、credential、shareの選択 |
| ツール固有設定の意味 | 設定値を外部ツールの動作へ反映 | TLS、リクエスト上限、calibration基準 |
| 未登録の生データ | ctxへ昇格するまでの出力や証跡の管理 | スキャナの生出力、ダウンロードファイル |

ツール所有のキャッシュは、削除して再生成できる最適化データであり、調査結果ではありません。キャッシュを削除すると再調査が発生する可能性はありますが、host、service、credential、note、logなどctxへ昇格済みの結果を削除してはいけません。

### 共有保存場所と共有形式は別

ctxは、ツールの状態をworkspaceへ追従させるためのデータディレクトリを提供することがあります。その配下に置かれたすべてのファイルが、ctxの公開スキーマになるわけではありません。

次の規則を適用します。

- ctx所有レコードは、存在する場合は文書化されたctxインターフェースから読み書きする
- ツール所有ファイルは、その形式を定義したツールだけが参照する
- 別のツールが他ツールの専用キャッシュを解析しない
- 再利用する調査結果はキャッシュファイルの共有ではなく、ctxの登録インターフェースから昇格する
- workspace削除時には、その配下のctx所有レコードとツール所有状態の両方が削除され得るため、ツールはworkspaceデータディレクトリを独立ストレージとして扱わない

### 設定は責任を分担する

設定の保存、キーの検証、ctxが公開するデフォルト値、永続化はctxが所有します。その設定を利用するツールは、値の運用上の意味を所有し、一貫して動作へ反映する責任を持ちます。

ほかのコマンドが利用しないデータには、ツール専用ディレクトリのprivate fileが適しています。ctxの設定キーを追加するのは、その値が利用者向けの共有ctx workflowに含まれる場合だけです。

### 調査結果の昇格

外部ツールの出力は、検証されて明示的に登録されるまでツール所有です。ctxへの昇格は次の順序で行います。

1. ツールが外部コマンドを実行し、自身で出力を解析する
2. ノイズ、不完全な出力、試走結果を除外する
3. 確定した結果をhostやcredentialなどctxの概念へ変換する
4. ctxが検証し、利用可能な根拠とともに共有レコードとして保存する

試走とcalibrationでは調査結果を自動登録しません。失敗または中断したコマンドでもcommand logを残す場合はありますが、部分的な出力を確定済みの共有調査結果として自動登録しません。

## 互換性を保つための規則

ctxの更新後も動作させる連携では、次を守ります。

- 文書化されたコマンド構文と終了コードを利用する
- 人間向けテーブルではなくJSONを利用する
- フォーマットバージョンを明示する
- メッセージの文言ではなく`success`と`error.code`を判定する
- JSONオブジェクトへ互換フィールドが追加されても処理できるようにする
- 必要な出力やバージョンが存在しない可能性がある場合は対応状況を確認する
- SQLiteスキーマへ依存しない

外部連携仕様は、現実的に維持できる限り互換性を保ちます。古い形式の維持が過度な負担になった場合は、フォーマットバージョンを変更し、破壊的変更をリリースノートへ明記した上で廃止することがあります。

## 公式コマンドとカスタムコマンド

公式の`x*`コマンドは、ツール固有の処理を直接実装できます。別バイナリからctxが所有するデータへアクセスする境界では、無理なく処理を表現できる場合に外部連携仕様を優先します。

利用者のカスタムコマンドにも同じ外部連携仕様を提供します。Goを使うこと、公式コマンドをコピーすること、このリポジトリへ取り込むことは要求しません。

### ctxを安全に起動する

ctxを別プロセスとして起動する同梱コマンドは、現在のAPTパッケージでは固定絶対パス`/usr/local/bin/ctx`を使用します。`PATH`から`ctx`を探索せず、シェルも起動せずに引数を分離して渡します。これにより、`PATH`の先に置かれた同名の実行ファイルへ調査情報やcredentialが渡ることを防ぎます。

共通の内部起動処理は相対パスを拒否し、テストでは実行パスとrunnerを個別に差し替えられます。ctxを別の場所へインストールする配布物は、依存するコマンドのビルド時に信頼する絶対パスを組み込みます。

```sh
go build -ldflags '-X req/internal/ctxexec.ExecutablePath=/opt/ctx/bin/ctx' ./cmd/xssh
```

`internal/ctxexec`は本リポジトリ内の実装補助であり、公開Go SDKではありません。利用者のカスタムコマンドでも、インストール時またはビルド時に選んだ信頼できるctxの絶対パスを直接実行し、引数を分離し、シェルの機能が明示的に必要な場合を除いてシェルを介さないことを推奨します。

## 現在の境界

連携対象は、具体的な利用実績がある能力に限定します。この文書は次を保証しません。

- 全ctxコマンドのJSON対応
- 汎用CRUD APIまたは調査結果保存API
- 人間向けテーブルの互換性
- SQLiteスキーマの互換性
- HTTP API
- 特定言語向けの公開SDK

新しい連携機能は、実際のコマンドによって不足する操作が確認された後に追加します。
